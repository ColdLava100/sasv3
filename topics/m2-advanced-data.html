🔗 4.1 Data Joins — Join vs Append

📌 The Critical Difference:
🔗 Data Join📥 Append
What it doesMatches rows → adds columnsAdds rows under matching columns
ResultWider table (more columns)Longer table (more rows)
Available in VA?✅ YES (Left, Right, Inner, Full)❌ NO — NOT available
AnalogyPutting two card decks side by side 🃏🃏Stacking one deck on top 🃏➡️🃏
⚠️ EXAM ALERT: Append is NOT available in Visual Analytics! Only Joins are available. This is a very common exam trick question ❗
🧠 Memory Trick: "Join = Join columns (sideways). Append = Add rows (downwards). Join is Just fine in VA. Append is Absent."

🔗 Join Types & Considerations

4 Join Types: Left, Right, Inner, Full
⚠️ Key facts:
• Join creates a temporary table — executed every time report opens (for every user)
• Stored in CASUSER caslib — visible only to creator
• Can join only 2 tables at a time — no self-joins ❌
• Only simple matching available — dates must have matching formats
• Temporary results can be very large — impacts concurrent user performance ⚠️

📅 4.2 Periodic Functions — The 5 Types

📌 What are Periodic Functions?
"Periodic functions aggregate values over time. Evaluated using time intervals." They're used for things like year-over-year comparisons, running totals, etc.
FunctionWhat it returnsAnalogy 🧠
CumulativePeriod Value for current period + ALL previous periods within a larger period Running total 🏃 — like a snowball rolling downhill, growing bigger
ParallelPeriod Value for the same period in a previous cycle (e.g., same quarter last year) Time travel ⏪ — go back to the same time last year
Period Value for a specific period of time SnapShot 📸 — just that one period, nothing else
PeriodWithDate Value for a specific, constant period (always the same dates) Fixed date 🔒 — always points to the same calendar period
RelativePeriod Value for a period relative to the current period (e.g., last 3 months) Moving window 🪟 — shifts based on "now"
🧠 Memory Trick:
• Cumulative = Current + all Coming before (running total)
• Parallel = Previous cycle (same spot, different year)
• Period = Point in time (just that one)
• PeriodWithDate = Permanent date (stays constant)
• Relative = Rolling window (moves with current date)

🔒 4.3 Suppress Function — Data Privacy

📌 What is Suppress?
The Suppress function hides (suppresses) aggregated data values for privacy. It's an Advanced Aggregated Function used when displaying aggregated data could reveal sensitive information about individuals.

🔓 Suppress — Without Complement

How it works:
Suppress(condition, measure, withoutComplement)

Example scenario: Hide "Number Tested" when the count is less than 5.

North High School has only 3 students tested → count is less than 5 → value is suppressed (shows *)

BUT: Total = 91. If you do 91 - (12+15+22+25+16) = 1, you can figure out North = 1.
With withoutComplement: The suppressed value can still be INFERRED from the total. The summary row makes inference possible.

🔑 In this mode: Only the values meeting the condition are hidden. If the total gives it away, that's acceptable.

🔒 Suppress — With Complement

How it works:
Suppress(condition, measure, withComplement)

Same scenario: Hide "Number Tested" when count is less than 5.

East High School also has a low count → East and North are both hidden (2 values hidden).

Why? If only North was hidden, you could calculate its value from the total. By hiding an additional value (East), you CANNOT tell whether any specific hidden school's value is high or low.

Total = 91, but now you don't know if East + North = some combination. You can't infer individual values.

🔑 In this mode: Additional values are suppressed to prevent inference. More values are hidden than strictly necessary.
ModeMeaningWhat gets hiddenCan you infer the value?
withoutComplement Without additional hiding Only values matching the condition ✅ Yes — if total is visible, you can calculate it
withComplement With additional hiding to prevent inference Condition values + extra values to obscure totals ❌ No — too many values hidden to calculate
🧠 Memory Trick:
• withoutComplement = "Without" = "What you see is what's hidden" (no extra hiding)
• withComplement = "Complement" = "Complete the hiding" (hide more to protect privacy)

"Without = you can figure it out (no extra hiding). With = you can't figure it out (extra values hidden)."
⚠️ EXAM ALERT: The key difference between withComplement and withoutComplement is that withComplement suppresses additional values beyond the condition to prevent inference from totals. If you see "Two values hidden" → that's withComplement ❗

❌ Your Quiz Mistakes — Explained Step by Step

❌ Mistake 1: Inner Join vs Left Join

Question: "To create one table that contains only the students that appear in BOTH tables, what kind of join?"
You answered: a) Left join ❌
Correct answer: c) Inner join ✅
Why not Left join? Left join keeps ALL rows from the LEFT table, even if they don't match the right. Ashley is in Math but not English — with Left join, Ashley would appear with NULL for English grade.

Why Inner join? Inner join = ONLY rows that match in BOTH tables. Only Robert and Rebecca appear in both English and Math tables → only those 2 students in the result.

📊 Visual:
Table A (English): Robert(95), Tom(90), Rebecca(97)
Table B (Math): Robert(75), Ashley(92), Rebecca(94)

Inner join → Robert + Rebecca (only ones in BOTH) ✅
Left join → Robert + Tom + Rebecca (all English, Ashley missing) ❌
Right join → Robert + Ashley + Rebecca (all Math, Tom missing) ❌
Full join → Robert + Tom + Rebecca + Ashley (everyone) ❌
🧠 Memory: "Inner = Intersection (only what's in BOTH). Left = Left table gets everything."

❌ Mistake 2: Aggregated Data Sources — Permanent or Temporary?

Question: "Which is NOT true about aggregated data sources?"
You answered: c) You can add a simple filter when creating ❌
Correct answer: d) Aggregated data sources are saved as a permanent table ✅
Why c is wrong? The statement "You can add a simple filter when creating an aggregated data source" is actually TRUE — you CAN add filters. So it's NOT the "not true" answer.

Why d is correct? Aggregated data sources are NOT permanent tables. They are temporary tables stored in your personal CASUSER library, generated every time the report opens. This is the FALSE statement → the "not true" answer.

✅ True statements about aggregated data sources:
• Can simplify visualizations and reduce processing time ✅
• Can create nested aggregated measure calculations ✅
• You CAN add a simple filter when creating ✅
❌ Are saved as permanent tables → FALSE, they're temporary!
🧠 Memory: "Aggregated data = Temporary, not permanent. Aggregated measures do NOT accept other aggregated measures."

❌ Mistake 3: FindChar — Text Function

Question: "Which text function can be used to find the position of a character within a text string?"
You answered: a) SearchChar ❌
Correct answer: d) FindChar ✅
Why SearchChar is wrong? "SearchChar" is NOT a real function in SAS VA. You made it up!

Why FindChar is correct? FindChar is in the Text (advanced) section of the expression editor.
FindChar(string, character) → returns the position (number) where the character first appears.

Example: FindChar("SAS Visual Analytics", "V") → returns 5 (V is the 5th character).

Other Text functions for reference:
• Substr(string, start, length) — Extract substring
• Trim(string) — Remove spaces
• Upcase/Lowcase — Change case
• ReplaceWord(string, old, new) — Replace words
• GetWord(string, position) — Get word at position
• FindChar(string, char) — Find position of character ✅
• Length(string) — Get string length
• Cat/Cats — Concatenate
🧠 Memory: "FindChar = Find the character's First position. There is NO function called SearchChar — it doesn't exist!"

❌ Mistake 4: AggregateCells — Running Total

Question: "Which aggregated operator can be used to calculate a running total over a category?"
You answered: b) CumulativeCells ❌
Correct answer: c) AggregateCells ✅
Why CumulativeCells is wrong? "CumulativeCells" is NOT a real function! You confused it with "CumulativePeriod" (periodic function) and made up a hybrid name.

Why AggregateCells is correct? "The AggregateCells operator applies an aggregate operation to a range of items in a group (for example, calculate a running total from the first row to the current row for a category data item)."

Key distinction — Don't confuse these:
FunctionCategoryWhat it does
CumulativePeriodPeriodicRunning total over TIME periods
AggregateCellsTabularRunning total over CATEGORY/group ✅
CumulativeCells❌NOT a real function!

So for a running total over a CATEGORY, use AggregateCells.
For a running total over TIME, use CumulativePeriod.
🧠 Memory: "Running total over Category = AggregateCells. Running total over time = CumulativePeriod."

❌ Mistake 5: AggregateTable — Final Aggregation Timing

Question: "In AggregateTable, I specify inner and outer (final) aggregation. When is the final aggregation applied?"
You answered: c) Always applied after inner ❌
Correct answer: a) Applied after inner, but ONLY if visualization has fewer group-by crossings ✅
The key nuance you missed:
The final/outer aggregation is NOT always applied. It's applied only when necessary — specifically, when the visualization has fewer group-by crossings than the inner aggregation.

Think of it like this:
• Inner aggregation: Calculates values at a detailed level (e.g., daily sales by product)
• Visualization: Shows only monthly totals (fewer group-by crossings)
• Final aggregation: Only then is it applied — to "squeeze" the detailed values into the fewer available cells

If the visualization has the same or more crossings than the inner aggregation, no final aggregation is needed!
🧠 Memory: "Final aggregation = Fallback. Applied Flexibly — only when visualization cells are too Few for the data."
